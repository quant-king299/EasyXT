# 复权方案简化说明

## 架构变更：从本地计算到API调用

### 旧方案（本地计算 + 缓存）
```
存储层: 4列原始数据
         ↓
查询层: 检查DuckDB缓存表
         ↓ 未命中
计算层: numpy向量化计算复权（需要分红数据）
         ↓
缓存层: 保存到DuckDB临时表
```

**代码量**: 316行
**依赖**: 需要集成分红数据源（tushare/akshare）
**复杂度**: 高（需自己实现复权算法）

---

### 新方案（直接调用QMT API）
```
存储层: 4列原始数据
         ↓
查询层:
  - 不复权 → 从DuckDB读取
  - 需要复权 → 调用QMT API（xtdata.get_market_data_ex）
```

**代码量**: ~200行
**依赖**: QMT在线环境
**复杂度**: 低（使用QMT官方算法）

---

### 新方案（API调用）
```python
def get_adjusted_data(self, stock_code, start_date, end_date, adjust_type, con):
    # 不复权：直接返回
    if adjust_type == 'none':
        return self._get_raw_data(...)

    # 需要复权：调用QMT API
    return self._get_from_qmt_api(...)

def _get_from_qmt_api(self, stock_code, start_date, end_date, adjust_type):
    dividend_type = ADJUST_TO_QMT_DIVIDEND_TYPE[adjust_type]

    data = self.xtdata.get_market_data_ex(
        stock_list=[stock_code],
        period='1d',
        start_time=start_date,
        end_time=end_date,
        dividend_type=dividend_type
    )

    return self._format_qmt_data(data)
```

## 性能对比

| 指标 | 旧方案（本地计算） | 新方案（API调用） |
|------|------------------|------------------|
| **代码行数** | 316行 | ~200行 ✅ |
| **首次查询** | 需计算（慢） | API调用（中） |
| **再次查询** | 从缓存读取（快） | API调用（中） |
| **准确性** | 取决于算法实现 | QMT官方100% ✅ |
| **依赖** | 分红数据源 | QMT在线 |
| **维护成本** | 高（需维护算法） | 低（无算法） |

## 优势总结

### 新方案优势 ✅
1. **代码简洁**: 减少37%代码量
2. **准确性高**: 使用QMT官方复权算法
3. **易维护**: 无需实现和维护复权算法
4. **无数据依赖**: 不需要集成分红数据源

### 新方案劣势 ⚠️
1. **依赖QMT**: 必须QMT在线运行
2. **查询速度**: 受API调用频率限制
3. **网络依赖**: 无法完全离线运行

---

## 使用建议

**选择新方案（API调用）如果：**
- ✅ 你的运行环境有QMT在线
- ✅ 需要准确的复权数据
- ✅ 希望代码简洁易维护

**保留旧方案（本地计算）如果：**
- ✅ 需要完全离线运行
- ✅ QMT环境不稳定
- ✅ 有可靠的分红数据源
