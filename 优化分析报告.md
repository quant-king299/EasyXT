 # EasyXT 项目深度优化分析报告
 
 ## 1. 风险级问题
 
 ### 1.1 149 个裸 `except:` 子句
 代码库中最危险的模式。所有异常被静默吞掉，导致线上 bug 极难追踪。
 
 ### 1.2 17,000+ 处 `print()` 输出
 主模块中约 8,800 处，没有日志级别控制、文件输出或结构化记录。
 
 ### 1.3 `app/` 与 `code_converter/` 代码完全重复
 - `app/py_converter/jq_to_easyxt.py` 与 `code_converter/converters/jq_to_easyxt.py` 一致
 - `app/py_converter/jq_to_ptrade_unified_v3.py` 与 `code_converter/converters/jq_to_ptrade_unified_v3.py` 一致
 - `code_converter` 是 git 子模块，`app/` 有自己的 `.git` 仓库
 
 ## 2. 架构级问题
 
 ### 2.1 两个独立配置管理系统
 - `core/config/config_manager.py` → `UnifiedConfigManager`
 - `core/data_manager/config.py` → `DataManagerConfig`
 
 ### 2.2 巨型单体文件过多
 超过 50KB 的文件有 24 个，最大的达 174KB。
 
 ### 2.3 `sys.path.insert()` 泛滥
 大量脚本通过 `sys.path.insert(0, ...)` 而非正确包结构管理路径。
 
 ### 2.4 模块加载时产生副作用
 `easy_xt/__init__.py` 在 import 时执行 monkey-patch、打印信息。
 
 ## 3. 代码质量与工程化
 
 ### 3.1 测试覆盖率低且不可靠
 - 核心模块只有 6 个独立脚本而非 pytest 用例
 - 没有 CI/CD 配置
 
 ### 3.2 硬编码路径
 如 `D:\StockData\stock_data.ddb` 散落各处。
 
 ### 3.3 版本约束不匹配
 `requires-python = ">=3.8"` 但代码使用了 3.9+ 特性。
 
 ### 3.4 依赖管理混乱
 三份依赖清单（pyproject.toml / requirements.txt / requirements_realtime.txt）重叠且差异不明确。
 
 ### 3.5 `.gitignore` 过于激进
 超过 500 行，`*test*.py` 排除了所有含 test 的文件。

 ## 7. 已修复的优化项
 
 ### 7.1 裸 `except:` 子句 (2026-06-24)
 
 **状态:** 已完成
 **范围:** 全仓库 149 处裸 `except:` 全部替换为具体异常类型
 
 处理策略:
 - `except (ValueError, TypeError)` — `strptime()`, `to_datetime()`, `to_numeric()`, `fromisoformat()`, `int()`, `float()` 等类型/值转换
 - `except (ImportError, AttributeError)` — 动态导入 (`import`, `from`) 语句
 - `except Exception` — 所有其他场景（数据库操作、网络请求、订阅行情、清理操作等）
 
 `except Exception:` 相比裸 `except:` 的关键改进：不会捕获 `SystemExit`、`KeyboardInterrupt`、`GeneratorExit`，避免程序在用户中断、系统退出时被静默吞掉。同时各类库抛出的异常都是 `Exception` 的子类，覆盖范围不受影响。
